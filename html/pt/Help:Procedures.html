Uma coleção de listas de verificação a serem usadas ao realizar alterações complexas a artigos ou operações de manutenção.

Mover uma seção dentro da mesma página

O movimento deve ser feito em uma única edição, que não altere nada mais.

  1. Abra o artigo em um editor.
  2. Recorte o texto a ser movido. Não salve a página agora, por exemplo, não realize a movimentação em duas edições, a primeira remoção da seção e a segunda colando-o, ou na segunda edição vai parecer que você é o autor, especialmente se o resumo da edição que a acompanha não estiver claro.
  3. Cole o texto recortado na nova posição.
  4. Adapte o nível de título se necessário, mas não faça outras adaptações ao conteúdo para o momento, do contrário as modificações não estarão visíveis no diff de edição resultante.
  5. Salve a página, escrevendo adequadamente o resumo de edição.
  6. Agora, você pode prosseguir com a edição do texto da seção normalmente, conforme necessário.

Dividir seção em uma nova subpágina

Esse procedimento é útil quando se está movendo uma seção de um artigo que se tornou muito longo para uma subpágina daquele artigo.

  1. Abra a seção do artigo de origem em um editor.
  2. Copia todo o conteúdo da seção.
  3. Abra a subpágina de destino em um outro editor.
  4. Cole o conteúdo copiado no editor de destino sem modificações.
  5. Salve a subpágina de destino com um resumo de edição como content split from [[Origin article#Section]]; certifique-se de incluir um link para a página original, ou vai parecer que você é o autor do conteúdo.
  6. No editor de origem, substitua o conteúdo divido com um link para a subpágina de destino seja deixando o título da seção no lugar, seja adicionando um link na caixa de Artigos relacionados do artigo.
  7. Salve a página de origem com um resumo de edição como content moved to [[Destination subpage]].
  8. Reabra a subpágina de destino em um editor.
  9. Categorize a subpágina de destino como artigo de origem.
  10. Adicione um link para o artigo de origem no topo; por exemplo See [[Origin article]] for the main article.
  11. Adapte os níveis de títulos da nova subpágina para iniciar do segundo nível.
    Dica: Essa etapa pode ser feita automaticamente com o plugin Wiki Monkey.
  12. Salve a subpágina de destina com um resumo de edição apropriado.

Outras etapas adicionais avançadas:

  • Verifique e corrija quaisquer links quebrados a seções em ambas páginas de origem e destinos, e em páginas fazendo link com página de origem.
Dica: Essa etapa pode ser feita automaticamente com o plugin Wiki Monkey.

Lidar com páginas de discussão após redirecionar uma página para outra

Se a página A tiver sido redirecionada para a página B, por exemplo após mesclar o conteúdo de A com a página B, e Talk:A existir:

  • Se Talk:B não existir, mova toda a Talk:A para Talk:B, deixando o MediaWiki redirecionar automaticamente Talk:A para Talk:B.
  • Se Talk:B existir:
    1. Mova as discussões ainda relevantes, se houver, de Talk:A para Talk:B.
    2. Certifique-se de que as discussões restantes em Talk:A, se houver, estejam fechadas.
      • Se Talk:A hospedar discussões que foram fechadas por menos tempo do que o especificado em Help:Discussão#Fechando uma discussão, marque Talk:A com Template:Redirect, de forma que a página seja redirecionada para Talk:B após o período de fechamento.
      • Se não, redirecione imediatamente Talk:A para Talk:B.

Corrigir redirecionamentos duplos

  1. Leia essa seção para entender o que são redirecionamentos.
  2. Confira Special:DoubleRedirects para ver se há algum.
  3. Por exemplo, se você vê Clientes pastebin (Editar) →‎ Aplicativos Comuns →‎ Lista de aplicativos, isso significa que Clientes pastebin redireciona para Aplicativos Comuns, e Aplicativos Comuns redireciona para Lista de aplicativos. Portanto, Clientes pastebin é um redirecionamento duplo.
  4. Para corrigir isso, edite Clientes pastebin e altere #REDIRECT [[Aplicativos Comuns]] para #REDIRECT [[Lista de aplicativos]] para pular o intermediário.
  5. Insira um resumo de edição tal como Fixed double redirect e salve.
Dica: Essa tarefa pode ser feita automaticamente com o plugin do Wiki Monkey.

Corrigir links quebrados de pacotes

O ArchWiki contém muitos links quebrados para pacotes não encontrados em repositórios oficiais ou no AUR, que é o resultado de pacotes serem mesclados, divididos ou removidos dos repositórios. Todas as páginas no espaço de nome principal são verificadas regularmente por um bot, que verifica todas as instâncias dos modelos AUR, Grp e Pkg, tenta para atualizá-las automaticamente e marcá-las com Template:Broken package link quando não for possível atualizá-las automaticamente.

Nota: caso deseje marcar manualmente um link de pacote como quebrado, use Template:Broken package link (Português), que é a versão traduzida deste último template acima ou, preferivelmente, corrija o link.

Para corrigir um link quebrado de pacote, não simplesmente remova referência aos pacotes do wiki, faça uma pesquisa primeiro:

  • Pesquise na base de dados de pacotes (pacman -Ss) e no AUR, visto que é possível que o pacote tenha sido mesclado/renomeado.
    • Para pacotes quebrados dos repositórios oficiais, você pode também pesquisar nos commits em https://gitlab.archlinux.org/archlinux/packaging/state/-/commits.
    • Para pacotes quebrados do AUR, você pode procurar nos arquivos da lista de discussão aur-request.
  • Se você estiver procurando por um arquivo específico, como, por exemplo, um binário que era parte de um pacote, pkgfile pode te ajudar nisso.
  • Se não tiver certeza, marque a página ou a seção com um template apropriado ao invés de remover completamente a referência ao pacote.

Para ajudar com atualizações manuais, cada modelo de "link de pacote quebrado" fornece uma dica:

  • "invalid number of template parameters" — Todos os templates AUR, Grp e Pkg aceitam exatamente um parâmetro, mas o textowiki especificou mais (ou nenhum). Na maioria dos casos, os parâmetros excedentes devem ser movidos para o texto em volta, ou removido se houver.
  • "replaced with [outro pacote]" — O pacote foi renomeado ou mesclado em outro pacote, que especifica o nome do pacote antigo no vetor replaces. Na maioria dos casos, o pacote antigo deve simplesmente ser substituído com o novo e o texto envolta adaptado adequadamente.
  • "package not found" — Dica padrão quando nenhuma das opções acima se aplicam.

Todas as páginas com links quebrados de pacotes podem ser rastreados em Category:Pages with broken package links. Há também uma página de relatório automático em User:Lahwaacz.bot/Reports/archpkgs.

Nota: O bot atualiza apenas links de pacotes, mas não o texto em volta, o que varia muito conforme o contexto. Por exemplo, no revision 308608 no link do AUR foi alterado para Pkg, mas o texto em volta ainda diz que o pacote está no AUR. Essas instâncias podem ser corrigidas e "a prova de futuro" simplesmente removendo a descrição em volta de onde o pacote está; veja também Help:Estilo#Instruções de gerenciamento de pacote. Atualmente, não temos como rastrear automaticamente esse tipo de problema, sugestões são bem-vindas.

Corrigir links quebrados de seções

Ocasionalmente, as páginas podem conter links de seção quebrados, que são o resultado de seções sendo renomeadas, mescladas, movidas ou excluídas das páginas. Todas as páginas no espaços de nome principal são verificadas regularmente pelo Lahwaacz.bot, que verifica todos os links e os marca com Template:Broken section link (ou sua versão em português, no caso de páginas traduzidas) quando os links de seção estão quebrados.

Para corrigir um link quebrado de seção, não simplesmente remova a referência aos links de seção do wiki, faça algumas pesquisas primeiro:

  • Olhando o histórico da página onde a seção está localizada, é possível que a seção tenha sido renomeada/mesclada/movida/excluída.
  • Se não tiver certeza, marque a seção com um modelo de status apropriado, em vez de remover completamente a referência à seção.

Todas as páginas com links de seção quebrados são rastreadas em Category:Pages with broken section links.

Corrigir links para páginas arquivadas

Quando uma página é arquivada, as diretrizes em ArchWiki:Arquivo#Como arquivar uma página definem que todos os links para a página devem ser removidos antes do arquivamento. Em alguns contextos (em particular para traduções, onde o contexto da frase não permite a simples remoção do link sem alterar a experiência do leitor), isso pode não ser feito ao arquivar a página.

Para corrigir um link para uma página arquivada, não basta remover o link da página sem antes fazer alguma pesquisa:

  • Ajuste a página substituindo a referência ao conteúdo desatualizado por sua substituição (por exemplo, Special:Diff/704983).
  • Para uma tradução em que você é fluente o suficiente para atualizar a página, corresponda ao que foi feito na página em inglês.
  • Se não tiver certeza, marque a seção com um template de status apropriado em vez de remover completamente o link.

Todas as páginas com links para páginas arquivadas são rastreadas em Category:Pages with links to archived pages.

Mover uma discussão para outra página de discussão

  1. Copie o texto da discussão para a página talk de destino, se certificando de adicionar, entre o novo título e o texto colado, uma nota como:
    ''[Moved from [[Página talk de origem#Título]]. -- ~~~~]''
  2. Marque com riscado o título na página talk de origem e substitua o conteúdo com uma nota como:
    ''[Moved to [[Página talk de destino#Título]]. -- ~~~~]''

Renomear uma categoria

  1. Mova a página de categoria da mesma forma como você moveria uma página normal, se assegurando de que um redirecionamento seja criado do título antigo para o novo. Isso só vai renomear a página de categoria em si, os membros da categoria não serão recategorizado.
  2. Recategorize todos os membros da categoria antiga para usar a nova.
    Dica: Isso pode ser feito automaticamente com o recategorize-over-redirect.py do wiki-scripts, o que se baseie no redirecionamento da categoria antiga para detectar o novo nome e, portanto, não esteja limitado a alterações na capitalização ou heurísticas similares.
  3. Atualize todos os links interlínguas.
    Dica: Isso pode ser feito automaticamente com o interlanguage.py do wiki-scripts ou o bot de plugin Wiki Monkey.
  4. Atualize todos os links que estejam apontando para a categoria antiga para que façam referência à nova.
    Dica: Isso pode ser feito automaticamente executando o plugin RegExp substitution do Wiki Monkey na página Special:WhatLinksHere da categoria antiga, com uma substituição como (\[\[|\{\{Related2?\|):[ _]*[Cc]ategory[ _]*:[ _]Nome[ _]*[Aa]ntigo[ _]*(#|\||\]\]|\}\}) -> $1:Category:Novo nome$2 (presumindo que o nome da categoria antiga seja "Category:Nome antigo").
  5. Marque a categoria antiga com Template:Archive, sem destruir o redirecionamento (a categoria antiga provavelmente ainda está vinculado pela Tabela de conteúdo).
    Dica: Se a categoria não possuir um histórico relevante, ela pode ser excluída por um administrador, após se certificar de que as etapas 2-4. tenham realmente sido realizadas.

Remover uma página inteira

  • Página nova, porém inapropriada:
    • se inapropriada por causa de spam ou outro conteúdo claramente malicioso (avaliação é feita por um Administrador), ela é removida imediatamente;
    • em outros casos, p. ex.: irrelevância para o Arch, o artigo deve ser:
      • marcado com Template:Merge; ou
      • marcado com Template:Redirect; ou
      • marcado para ser movido como uma subpágina da página User do autor com Template:Move (ou movido imediatamente).
  • Artigo antigo se tornou obsoleto:
    1. marque com um dos seguintes, em ordem de preferência:
      • Template:Merge;
      • Template:Redirect;
      • Template:Archive;
    2. aguardo por melo menos 7 dias;
    3. na ausência de discussões de objeção, ou quando essas eventualmente forem resolvidas em favor da remoção, realize a ação proposta;
      • ao redirecionar, considere seguir #Lidar com páginas de discussão após redirecionar uma página para outra e #Corrigir redirecionamentos duplos;
      • ao arquivar, siga as instruções em ArchWiki:Arquivo.

Patrulhamento

A verificação das mudanças recentes pode ser feita por todo mundo. Membros da Equipe de Manutenção, além disso, também podem marcar edições e páginas como patrulhadas. Esta seção é principalmente sobre as coisas que todo mundo pode fazer.

Tenha em mente que patrulhar as mudanças recentes obviamente requer um compromisso mais constante, enquanto consertar outras coisas é muito mais flexível e pode ser feito sempre que você tiver tempo.

Patrulhamento de mudanças recentes

Existem duas maneiras principais de patrulhar as mudanças recentes:

  • Visitar Special:RecentChanges em intervalos regulares, verificando todas as edições que foram feitas desde a última visita.
  • Inscrever-se ao feed Atom de Special:RecentChanges e verificar as edições assim que você tiver algum tempo.

Para cada edição ou grupo de edições feitas na mesma página, você deve avaliar se é questionável, conforme sua experiência e conhecimento, também levando em consideração a lista de problemas mais frequentes.

  • Se você acha que a edição requer uma solução rápida que você pode executar imediatamente, basta fazê-lo. Isso se aplica especialmente para pequenos problemas de estilo e erros de escrita.
  • Se, em vez disso, a edição for questionável, mas você não puder corrigi-la, você deve verificar se ela já foi sinalizada com o modelo apropriado:
    • Caso contrário, adicione um modelo apropriado à seção apropriada descrevendo o problema.
    • Caso já haja uma discussão sobre a edição, veja se você pode adicionar detalhes úteis à observação ou discussão que a acompanha.
Dica: Torne o patrulhamento mais fácil seguindo os seguintes passos:
  • Habilite a configuração Agrupar alterações por páginas nas mudanças recentes e nas páginas vigiadas sob Preferências > Mudanças recentes > Opções avançadas.
  • Para seguir seus artigos vigiados, usando um leitor de feed, use o link Atom na coluna da esquerda de sua página de páginas vigiadas.

Edições por bots

O MediaWiki não exibe as edições feitas por bots nas mudanças recentes por padrão. Pode ser desejável verificar quando um dos bots modificou as páginas, pois isso pode indicar que as alterações são necessárias.

Os bots sinalizam links quebrados de pacotes, links quebrados de seções e links inativos (dead links).

É altamente recomendado corrigir as coisas sinalizadas assim que elas aparecerem antes de serem enterradas sob uma tonelada de mudanças. Isso também se aplica a links inativos, que geralmente são recursos externos.

Problemas comuns e soluções

Nota: Você sempre pode pedir ajuda à Equipe de Manutenção.

Abuso

Felizmente, spam e outras coisas que violam o Código de conduta são bastante incomuns, mas ainda acontecem ocasionalmente.

A tarefa mais importante também se aplica aqui. Certifique-se de combater o abuso imediatamente:

  1. Em primeiro lugar, desfaça todo o dano.
  2. Entre em contato com a Equipe de Manutenção. Você também pode fazer isso juntando-se ao canal de IRC do ArchWiki e mencionando os administradores, que geralmente são todos operadores de canal.

Se houver uma onda de abuso e desfazer o dano levar muito tempo, denuncie primeiro.

Relacionados a conteúdo

  • Remoção de conteúdo útil: desfaça ou contate o autor.
  • Modificação ou remoção de conteúdo não explicado: contate o autor; desfaça se não houver reação.
  • Modificação grande (geralmente em uma única grande edição) sem explicação suficiente: contate o autor.

Relacionados a estilo

  • Assinatura, créditos, observações pessoais em artigos: desfaça ou mova à página talk.
  • Cabeçalho inicial do nível 1: mova todas seções 1 nível acima.
  • Novo artigo não categorizado: adicione uma categoria e corrija o cabeçalho.
  • Uso impróprio de modelos: corrija de acordo com Help:Estilo.
  • Adição de instruções de instalação: desfaça ou altere para cumprir com Help:Estilo.

O recurso de patrulhamento do MediaWiki

Nota: Este recurso só pode ser usado por membros da Equipe de Manutenção.

Marcar alterações como patrulhadas é uma maneira muito útil de evitar fazer as coisas duas vezes quando não for necessário. Todos os membros da Equipe de Manutenção são incentivados a usar esse recurso para economizar o tempo dos outros.

Às vezes, especialmente quando alguém não tem experiência com um tópico, não fica claro se deve marcar a edição como patrulhada ou não. Não marcar uma edição como patrulhada significa que é mais provável que outros membros da Equipe de Manutenção vejam a alteração. Aqui estão algumas dicas que podem ajudar a patrulhar mais mudanças:

  • Erros de digitação, gramática ou correções de idioma geralmente são muito fáceis de validar. Cuide deles primeiro, se possível.
  • Adições às páginas de discussão podem ser marcadas como patrulhadas se fizerem sentido (apropriado para o tópico, por exemplo, discutir jogos em uma página sobre o archiso não é apropriado).
  • As páginas do usuário podem conter qualquer coisa, desde que não viole nenhuma regra. Infelizmente, eles estão basicamente isentos da maioria das diretrizes de estilo.
  • Qualquer desenvolvedor pode fazer qualquer coisa na DeveloperWiki.
  • As traduções são difíceis de verificar se o idioma em questão não for conhecido. Pode-se usar DeepL para verificar se a tradução está aproximadamente correta. Verificar traduções de usuários muito novos ajuda a encontrar edições de baixa qualidade e vandalismo.
  • Erros acontecem. Se um usuário perceber um erro deles e desfazer sua própria edição, não há problema em marcar os dois como patrulhados.
  • Marque as edições que foram revertidas por um membro da Equipe de Manutenção como patrulhadas, pois quem reverteu a edição provavelmente esqueceu.
  • Novas páginas podem estar incompletas e/ou ter problemas de estilo. Marque-as como patrulhadas se ela se encaixar no ArchWiki. Considere vigiar esta página e se certificar de cuidar dela se ela for abandonada.

Todos esses pontos implicam que a mudança não deve violar o código de conduta ou as regras descritas em ArchWiki:Contribuindo.

Outros

Há outras coisas que precisam ser tratadas:

  • Verifique a lista de contas recém-criadas para ver se alguma delas já foi editada. As edições feitas por editores muito novos devem sempre ser verificadas, pois o editor pode não estar familiarizado com todas as diretrizes ainda.
  • Certifique-se de que quaisquer adições às páginas de discussão sejam assinadas (Para páginas traduzidas, veja Template:Não assinou).
    • Às vezes, os usuários não estão cientes do botão Adicionar tópico em uma página de discussão. Certifique-se de que as novas discussões sejam colocadas na parte inferior e tenham um nome de seção apropriado.
    • Se o resumo da edição se parecer com →Alguma seção: nova seção, o usuário usou o botão Adicionar tópico.
    • Se um usuário adicionar consistentemente novas seções manualmente, sinta-se à vontade para lembrá-lo de que existe o botão "Adicionar tópico" mais conveniente.
  • Exclusões e edições com diferenças muito grandes devem sempre ser investigadas.
  • Novas páginas também merecem uma olhada, melhorá-las e corrigir problemas de estilo é um bom exemplo e ajuda o autor.
  • Certifique-se de que as edições que modificam as tabelas não as quebram, por exemplo, inserindo uma coluna perdida.
  • Edições sem um resumo de edição adequado precisam de atenção especial. Há também Template:Editsum se o usuário não estiver usando os resumos de edição corretamente.
  • Desfazer é uma ação comum, mas ainda deve ser verificado.

Solução de relatórios

Veja ArchWiki talk:Requests.

  • Se você acha que pode corrigir o relatório, basta fazer isso e remover os modelos apropriados. Leia também #Problemas comuns e soluções
  • Caso contrário, se você sentir que é melhor entrar em contato com o(a) autor(a) da edição questionável, escreva-lhe uma mensagem em sua página de discussão ou enviar-lhe um e-mail para solicitar uma explicação ou discutir mais.
Dica:
  • De preferência para tentar corrigir os relatórios mais antigos
  • De preferência para corrigir problemas relacionados a conteúdo àqueles relacionados a estilo.
  • Considere usar um assistente de edição (p. ex., configuração de editor do Wiki Monkey) para resolver automaticamente alguns problemas de estilo comuns.
